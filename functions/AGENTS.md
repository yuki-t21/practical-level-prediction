# Cloud Run Functions 開発エージェント向け指示書 (AGENTS.md)

本ディレクトリ (`functions/`) 配下の Python Cloud Run Functions の変更・開発を行うエージェントは、以下のプラクティスを必ず遵守してください。

---

## 1. プロジェクト構成

本ディレクトリには、以下の 2 つの関数プロジェクトが配置されています。
- **`import-skill-check/`**: Excel データを縦持ちに変換し、BigQuery へ `WRITE_TRUNCATE` で上書きインポートする関数。
- **`export-prediction/`**: 推論対象ユーザーリストに対し、BQML モデルを用いたバッチ予測および SHAP 重要度の平坦化を行い、サービス名を付与した Excel 予測レポートを出力する関数。

---

## 2. uv を用いたパッケージ・仮想環境管理

本プロジェクトの Python 依存パッケージおよび仮想環境の管理には、高速パッケージマネージャーである **`uv`** を使用します。
レガシーな `pip` でのパッケージ操作や、直接的な `venv` 管理は行わないでください。

### ① パッケージの追加・更新
各関数ディレクトリ（`import-skill-check/` または `export-prediction/`）の配下で、以下のコマンドを使用してパッケージを追加・更新します。
これにより、`pyproject.toml` および `uv.lock` が自動的に同期されます。

```bash
# パッケージの追加
uv add <package_name>

# 開発用パッケージ (pytest 等) の追加
uv add --dev <package_name>

# 依存関係の同期
uv sync
```

### ② 依存関係のバージョン固定義務化 (重要)
- 本番用パッケージ (`dependencies`) および開発用パッケージ (`dependency-groups.dev`) の双方において、**依存関係のバージョンは必ず `==` を用いて厳密に固定してください。**
- `uv add` を実行した際、`pyproject.toml` に `>=` や `^` などで範囲指定される場合がありますが、その場合は手動で `==<バージョン>` に書き換え、`uv sync` で同期し直してください。
- 範囲指定による意図しないアップデートや、デプロイビルド時・CI/CD 時の不整合を防ぐため、厳格なバージョン固定は必須要件です。

---

## 3. 単体テスト (pytest) の義務化

ロジックの修正や新規機能の追加を行う際は、**必ず単体テスト（`test_main.py`）を整備・拡張してください。**
また、デプロイやコミットを行う前に、すべてのテストケースがパスすることをローカルで検証してください。

### テスト実行コマンド (各関数ディレクトリ配下で実行)
```bash
uv run pytest
```

---

## 4. インポート用スキーマの二重管理について (重要)

インポートを担当する `skill_check_results` と `service_master` の 2 テーブルについては、以下の 2 箇所でスキーマ情報（カラム名、説明、データ型）をそれぞれ定義しています。

1. **Dataform定義**: `dataform/definitions/sources/` 配下の SQLX ファイル
2. **Pythonインポーター定義**: `functions/import-skill-check/schemas/` 配下の JSON ファイル

### 開発時のルール
- デプロイパッケージの疎結合性（Cloud Run Functions が `dataform/` フォルダを必要としない構成）および Python 側ユニットテスト（`test_main.py`）のローカル実行独立性を維持するため、**意図的にこの二重管理構成を採用しています。**
- スキーマ変更時（カラムの追加や説明文の更新等）は、**必ず Dataform 側の SQLX ファイルと Python 側の JSON ファイルの双方を手動で同時に更新してください。**

## 5. コード品質管理 (Black, Flake8 & cSpell)

コードの品質保持、構文・エラーチェック、およびスペルミスの防止のため、フォーマッタである **`black`**、静的解析（リンター）ツールである **`flake8`**、およびスペルチェッカーである **`cspell`** を導入しています。

### ① Black による自動フォーマット
各関数ディレクトリ配下で以下のコマンドを実行し、コードの整形を行います。

```bash
# フォーマットの実行
uv run black .

# フォーマット違反がないかのチェック
uv run black --check .
```

### ② Flake8 による静的解析（コードチェック）
各関数ディレクトリ配下で以下のコマンドを実行し、スタイルガイド準拠や構文エラーのチェックを行います。
※Black と衝突する一部ルール（E501 行長エラーなど）は、リポジトリルートの `.flake8` にて無視する設定となっています。

```bash
# 静的解析の実行
uv run flake8 .
```

### ③ cSpell によるスペルチェック
リポジトリのルートディレクトリで以下のコマンドを実行し、スペルミスがないか確認します。
プロジェクト独自の許可ワードや除外設定はルートの `.cspell.json` にて管理されています。

```bash
# スペルチェックの実行
npx cspell "functions/**/*"
```
